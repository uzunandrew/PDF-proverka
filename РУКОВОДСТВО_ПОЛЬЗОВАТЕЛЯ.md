# Audit Manager — Как это работает (простым языком)

---

## Что это такое?

Система автоматической проверки проектной документации по электроснабжению.

**Вход:** PDF-файл проекта (чертежи + пояснительная записка)
**Выход:** Список замечаний с привязкой к нормам РФ + Excel-отчёт

**Кто проверяет:** Claude AI (читает текст и смотрит на чертежи)

---

## Общая схема: от PDF до отчёта

```
  PDF-файл
     │
     ▼
┌─────────────┐
│ ПОДГОТОВКА  │ ← process_project.py
│             │
│ 1. Текст    │ → extracted_text.txt
│ 2. Нарезка  │ → tiles/page_07/*.png
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ АНАЛИЗ      │ ← Claude AI (несколько сессий)
│ ЧЕРТЕЖЕЙ    │
│             │
│ Пакет 1     │ → tile_batch_001.json
│ Пакет 2     │ → tile_batch_002.json
│ ...         │ → tile_batch_NNN.json
│             │
│ Слияние     │ → 02_tiles_analysis.json
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ ОСНОВНОЙ    │ ← Claude AI (одна большая сессия)
│ АУДИТ       │
│             │
│ Текст + Чер-│
│ тежи + Нормы│ → 03_findings.json (ГЛАВНЫЙ ФАЙЛ)
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ ОТЧЁТ       │ ← generate_excel_report.py
│             │
│ Excel-файл  │ → audit_report_YYYYMMDD.xlsx
└─────────────┘
```

## Пошаговый алгоритм

### ШАГ 1. Создание проекта

**Что делаешь ты:**
1. Создаёшь папку в `projects/`, например `projects/МойПроект/`
2. Кладёшь туда PDF-файл проекта (назвать `document.pdf`)
3. Создаёшь файл `project_info.json` с минимальной информацией:

```json
{
  "project_id": "МойПроект-ЭМ",
  "name": "МойПроект-ЭМ",
  "section": "EM",
  "description": "Электроснабжение жилого дома",
  "pdf_file": "document.pdf"
}
```

**Результат:** Папка с PDF и конфигом готова.

---

### ШАГ 2. Подготовка данных (process_project.py)

**Что происходит:**
Скрипт `process_project.py` делает две вещи:

#### 2а. Извлечение текста из PDF
- Берёт PDF → вытаскивает весь текст со всех страниц
- Если в PDF использованы CAD-шрифты (ISOCPEUR) и текст "сломан" → автоматически запускает OCR (распознавание через Tesseract)
- Результат: файл `_output/extracted_text.txt`

#### 2б. Нарезка чертежей на тайлы
- PDF содержит большие чертежи (А1, А2, А3)
- Claude не может прочитать мелкие надписи на полном листе А1
- Поэтому каждый лист **нарезается на кусочки (тайлы)**
- Маленькие страницы (А4 — пояснительная записка) пропускаются
- Большие страницы режутся на сетку, например 2 ряда × 4 колонки = 8 тайлов

**Пример:**
```
Лист 7 (А2, однолинейная схема) → 8 тайлов:
  page_07_r1c1.png  page_07_r1c2.png  page_07_r1c3.png  page_07_r1c4.png
  page_07_r2c1.png  page_07_r2c2.png  page_07_r2c3.png  page_07_r2c4.png
```

Между соседними кусочками есть **перекрытие 8-12%**, чтобы надписи на границе не потерялись.

**Профили качества нарезки:**

| Профиль   | Размер тайла | Скорость | Когда использовать        |
|-----------|-------------|----------|--------------------------|
| draft     | ~1800 px    | Быстро   | Черновая проверка         |
| standard  | ~1500 px    | Средне   | Обычная работа (по умолчанию) |
| high      | ~1200 px    | Медленно | Мелкие надписи, спецификации |

**Как запустить:**
```bash
python process_project.py projects/МойПроект
```

**Результат:**
```
projects/МойПроект/_output/
├── extracted_text.txt           ← весь текст
└── tiles/
    ├── page_07/
    │   ├── page_07_r1c1.png     ← кусочек чертежа
    │   ├── page_07_r1c2.png
    │   ├── ...
    │   └── index.json           ← координаты каждого кусочка
    ├── page_09/
    └── page_14/
```

### ШАГ 3. Разбиение на пакеты (generate_tile_batches.py)

**Зачем:** В проекте может быть 100-1000+ тайлов. Одна сессия Claude не может просмотреть все — ограничение по контексту. Поэтому тайлы делятся на **пакеты по ~10 штук**.

**Правило:** Одна страница чертежа НИКОГДА не разбивается между разными пакетами. Все тайлы одной страницы идут вместе.

**Пример:**
```
Пакет 1: page_07 (8 тайлов) + page_08 (2 тайла из верхнего ряда)
Пакет 2: page_08 (оставшиеся 10 тайлов)
Пакет 3: page_09 (8 тайлов)
...
```

**Как запустить:**
```bash
python generate_tile_batches.py projects/МойПроект
```

**Результат:** `_output/tile_batches.json` — план, какие тайлы в каком пакете.

---

### ШАГ 4. Анализ чертежей по пакетам (Claude AI)

**Что происходит:**
Для каждого пакета запускается отдельная сессия Claude CLI.

Claude получает:
- Список тайлов пакета (PNG-картинки)
- Шаблон задачи (`.claude/tile_batch_task.md`)
- Контекст проекта

Claude делает:
- Смотрит на каждый тайл (картинку чертежа)
- Читает надписи: номиналы автоматов, сечения кабелей, марки оборудования
- Ищет ошибки: несоответствия, дубли, пропуски
- Записывает найденные проблемы

**Результат каждого пакета:** `tile_batch_001.json`, `tile_batch_002.json`, ...

**После всех пакетов — слияние:**
```bash
python merge_tile_results.py projects/МойПроект
```

**Итог:** `_output/02_tiles_analysis.json` — ВСЕ находки со ВСЕХ чертежей.

---

### ШАГ 5. Основной аудит (Claude AI)

**Что происходит:**
Одна большая сессия Claude, которая:

1. **Читает текст** (`extracted_text.txt`) → ищет нормативные ссылки, параметры, расчёты
2. **Читает результаты анализа чертежей** (`02_tiles_analysis.json`) → визуальные находки
3. **Сверяет с нормами** (`norms_reference.md` + поиск в интернете)
4. **Сводит всё вместе** → единый список замечаний

Для каждого замечания указывает:
- **Что не так** (проблема)
- **Где** (номер листа, тайл)
- **Какая норма нарушена** (конкретный пункт СП/ГОСТ/ПУЭ)
- **Как исправить** (рекомендация)
- **Насколько серьёзно** (категория)

**Результат:** `_output/03_findings.json` — **ГЛАВНЫЙ ФАЙЛ** со всеми замечаниями.

---

### ШАГ 6. Верификация норм (опционально)

**Зачем:** Claude может сослаться на устаревший ГОСТ или неправильный пункт СП. Этот шаг проверяет каждую нормативную ссылку через интернет.

**Что проверяет:**
- Документ действует или отменён?
- Есть ли замена?
- Правильный ли номер пункта?

**Результат:** Обновлённые замечания с проверенными ссылками на нормы.

---

### ШАГ 7. Генерация Excel-отчёта

**Что делает:** Берёт `03_findings.json` и создаёт красивый Excel-файл.

**Содержимое:**
- **Лист "СВОДКА"** — статистика: сколько замечаний какой категории в каждом проекте
- **Отдельный лист для каждого проекта** — таблица со всеми замечаниями

**Как запустить:**
```bash
python generate_excel_report.py
```

**Результат:** `audit_report_20260301_1430.xlsx`

## JSON-конвейер (4 этапа)

Каждый этап читает результат предыдущего и добавляет свою часть:

```
00_init.json          →  "Что у нас есть? PDF, тайлы, текст..."
     │
     ▼
01_text_analysis.json →  "В тексте нашли: нормы, параметры, подозрительные места..."
     │
     ▼
02_tiles_analysis.json → "На чертежах нашли: ошибки в номиналах, дубли, пропуски..."
     │
     ▼
03_findings.json      →  "ИТОГО: 30 замечаний. 3 критических, 7 существенных..."
```

**Золотое правило:** Если `03_findings.json` существует — ответы на вопросы берём ОТТУДА, а не перечитываем весь проект заново.

## Веб-интерфейс (Audit Manager)

**Адрес:** http://localhost:8080

### Главная страница — Дашборд
- Список всех проектов
- Статус каждого (зелёные кружки = этапы пройдены)
- Счётчики замечаний по категориям (цветные кружки)

### Для каждого проекта:
- **"Открыть"** — детальная страница проекта
- **"Замечания"** — таблица всех замечаний с фильтрами

- Оранжевый индикатор "Анализ тайлов: пакет 5/27 (18%)" — видно прогресс
- WebSocket — обновление в реальном времени без перезагрузки
- Если WebSocket отвалился — переподключается автоматически (до 5 попыток)

| `DELETE /api/audit/{id}/cancel`    | Отменить аудит                   |

## Нормативная база

Файл `norms_reference.md` — справочник действующих норм.

**Приоритет документов (от высшего к низшему):**
1. Федеральные законы (ФЗ-384, ФЗ-123)
2. Технические регламенты
3. Своды правил (СП) из обязательного перечня
4. Своды правил из добровольного перечня
5. ГОСТ
6. ПУЭ (в части, не противоречащей СП)

**Важно про ПУЭ:** ПУЭ-7 НЕ зарегистрирован Минюстом → применяется добровольно. При ссылке на ПУЭ всегда даётся параллельная ссылка на соответствующий СП.

## Что проверяет система на чертежах

### Однолинейная схема (Лист 7):
- Номиналы автоматических выключателей
- Наличие УЗО/УЗДП на нужных группах
- Трансформаторы тока (номинал, класс точности)
- Наличие АВР для важных потребителей
- Соответствие кабелей спецификации

### Схемы щитов (Лист 9):
- Совпадение групп со спецификацией
- Сечения кабелей для каждой группы
- Дубли обозначений (одинаковые имена у разных линий)
- Наличие PE-шины

### Планы (Листы 8, 11, 13):
- Цвета кабелей (синий = обычные, красный = огнестойкие)
- Все щиты с плана есть в спецификации
- Трассы огнестойких кабелей не через опасные зоны

### Узел ввода (Лист 14):
- Способ прокладки вводных кабелей
- Защита от механических повреждений
- Сечения соответствуют мощности

## Частые вопросы

**В: Зачем нарезать чертежи на тайлы?**
О: Claude видит картинку целиком, но на листе А1 мелкие надписи (номиналы 2А, сечение 1.5мм²) сливаются. Тайл — увеличенный фрагмент, где всё читаемо.

**В: Почему пакетный анализ, а не всё за раз?**
О: У Claude есть ограничение по контексту. 200 картинок за раз он не обработает. Пакеты по 10 штук — оптимальный баланс.

**В: Что такое "зомби-задача"?**
О: Задача, которая запустилась, но зависла (Claude упал, интернет пропал). Система автоматически убивает задачи без активности > 10 минут.

**В: Почему MD-файл менее надёжен, чем PDF?**
О: MD создаётся внешней программой через OCR. OCR может неправильно распознать символы: например `ППнг2(А)-HF` вместо правильного `ППГнг(А)-HF`.

**В: Можно ли запустить аудит только одного проекта?**
О: Да, на дашборде нажми "Открыть" → на странице проекта кнопка "Полный аудит".

**В: Что если аудит завис?**
О: Нажми "Отменить" на дашборде. Или перезапусти сервер — зомби-задачи удалятся автоматически при старте.
