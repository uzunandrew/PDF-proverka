## Направления оптимизации

### 1. ЗАМЕНА НА БОЛЕЕ ДЕШЁВЫЕ АНАЛОГИ
Проанализируй спецификации. Предложи аналоги с сопоставимыми характеристиками по более низкой цене. Укажи: исходную позицию, аналог, разницу в цене (%), обоснование эквивалентности.

### 2. УДЕШЕВЛЕНИЕ / УСКОРЕНИЕ МОНТАЖА
Найди позиции, где выбор материала усложняет монтаж. Предложи замены: более технологичные материалы, сборные соединения вместо сварки, быстромонтируемые крепежи. Указывай экономию в стоимости и человеко-часах.

### 3. УПРОЩЕНИЕ КОНСТРУКТИВА
Оцени компоновочные решения (обвязки, коллекторы, щиты, крепёж, проходки). Предложи упрощение конструкции, унификацию деталей, сокращение соединений или переход на заводские модули.

### 4. ОПТИМИЗАЦИЯ ЖИЗНЕННОГО ЦИКЛА
Решения дороже при строительстве, но с экономией на горизонте 10-25 лет: энергоэффективность, ремонтопригодность, увеличение межсервисных интервалов, снижение стоимости ЗИП.

## Правила анализа

- Все замены должны соответствовать действующим нормам (СП, ГОСТ, ТР ТС)
- НЕ предлагать замены, снижающие надёжность или пожарную безопасность
- При сомнениях — `"status": "требует проверки"`
- Приоритет: сначала крупные позиции с максимальным эффектом
- **Согласованность с замечаниями аудита:** если `03_findings.json` содержит замечание о несоответствии позиции нормам — НЕ предлагай для неё дешёвый аналог. Вместо этого можешь отметить что позиция требует замены по нормативным причинам (status: "обязательное исправление")
- **Вендор-лист обязателен:** при замене оборудования предлагай ТОЛЬКО производителей из вендор-листа. Если текущий производитель уже есть в вендор-листе — можно предложить другого из того же списка. Если текущего нет в списке — это само по себе замечание (status: "обязательное исправление")
- **Количество позиций определяется ТОЛЬКО по факту анализа** — столько, сколько реально найдено. Может быть 3, может 30. НЕ подгоняй под круглое число
- **Распределение по типам** — сколько получилось в каждой категории, а не равномерно
- **Процент экономии (savings_pct)** — только если можешь обосновать расчётом (разница цен, сокращение трудозатрат). Если не можешь посчитать — ставь `0` и поясни в `proposed`
- **estimated_savings_pct** — средневзвешенная оценка по крупным позициям, НЕ выдумывай круглое число. Если нет данных для расчёта — ставь `0`

## Порядок работы

1. Прочитай MD-файл целиком — тип объекта, ТЭП, нагрузки, оборудование
2. Прочитай `02_blocks_analysis.json` (если есть)
3. Прочитай `03_findings.json` (если есть) — чтобы не противоречить замечаниям аудита
4. Сверься с вендор-листом выше — допустимые производители для замен
5. Проанализируй спецификации — основной источник позиций для оптимизации
6. Сформируй оптимизационный лист (замены только из вендор-листа)
7. Запиши результат

## Выходной файл

ЗАПИСАТЬ через Write: `{OUTPUT_PATH}/optimization.json`

**СТРОГО СЛЕДУЙ СХЕМЕ. Каждый item — плоский объект с этими полями:**

```json
{
  "meta": {
    "project_id": "{PROJECT_ID}",
    "project_name": "",
    "analysis_date": "YYYY-MM-DD",
    "total_items": 0,
    "by_type": {
      "cheaper_analog": 0,
      "faster_install": 0,
      "simpler_design": 0,
      "lifecycle": 0
    },
    "estimated_savings_pct": 0,
    "top3_summary": ""
  },
  "items": [
    {
      "id": "OPT-001",
      "section": "",
      "current": "",
      "proposed": "",
      "type": "cheaper_analog",
      "savings_pct": 0,
      "timeline_impact": "без изменений",
      "risks": "",
      "status": "предложение",
      "norm": ""
    }
  ]
}
```

**Типы значений:** `total_items`, `by_type.*`, `estimated_savings_pct`, `savings_pct` — числа (number). Остальное — строки. Пустое поле: `""` для строк, `0` для чисел.

### Поля

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | `OPT-001`, `OPT-002`... |
| `section` | string | Откуда позиция: `"Спецификация лист 16"` |
| `current` | string | Что есть сейчас (макс 200 символов) |
| `proposed` | string | Что предлагается (макс 200 символов) |
| `type` | string | `cheaper_analog` / `faster_install` / `simpler_design` / `lifecycle` |
| `savings_pct` | number | Экономия % (0-100). Если не числовая — `0` |
| `timeline_impact` | string | `"без изменений"`, `"+2 дня"`, `"-3 дня"` |
| `risks` | string | Описание рисков. Нет рисков → `"минимальные"` |
| `status` | string | `"предложение"` / `"требует проверки"` / `"обязательное исправление"` |
| `norm` | string | Нормативная ссылка или `""` |

### Запреты

- НЕ создавай вложенные объекты — только плоские строки
- НЕ переименовывай поля
- НЕ добавляй поля, которых нет в схеме
- НЕ используй `null` — используй `""` или `0`
- НЕ создавай MD-отчёт — только `optimization.json`
