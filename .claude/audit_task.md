# ЗАДАЧА АУДИТА — КОНВЕЙЕРНЫЙ РЕЖИМ (JSON PIPELINE)

## Режим работы
Работай АВТОНОМНО. Не задавай вопросов. Не жди подтверждений.
Каждый этап ОБЯЗАН записать JSON-файл перед переходом к следующему.
Следующий этап читает JSON предыдущего — НЕ сканирует контекст заново.

## Структура конвейера

```
СТАРТ
  │
  ▼
[00] Инициализация ──────────────────────► ПИШЕТ: _output/00_init.json
  │
  ▼
[01] Анализ текста PDF ──────────────────► ПИШЕТ: _output/01_text_analysis.json
  │   ЧИТАЕТ: 00_init.json + extracted_text.txt
  ▼
[02] Анализ тайлов чертежей ─────────────► ПИШЕТ: _output/02_tiles_analysis.json
  │   ЧИТАЕТ: 01_text_analysis.json (список items_to_verify_in_tiles)
  ▼
[03] Свод замечаний ─────────────────────► ПИШЕТ: _output/03_findings.json
  │   ЧИТАЕТ: 01_text_analysis.json + 02_tiles_analysis.json
  ▼
[04] Финальный отчёт ────────────────────► ПИШЕТ: _output/audit_results_YYYYMMDD.md
      ЧИТАЕТ: 03_findings.json (только его, ничего больше)
КОНЕЦ
```

ПРАВИЛО: ВСЕГДА выполняй все этапы. Перезаписывай существующие JSON-файлы.
Каждый запуск аудита должен создавать СВЕЖИЕ результаты на основе актуальных данных.
Не пропускай этапы, даже если выходные файлы уже существуют.

## ЭТАП 00 — ИНИЦИАЛИЗАЦИЯ (включая ОБЯЗАТЕЛЬНУЮ нарезку тайлов)

Действие: прочитать project_info.json, подготовить ВСЕ данные для анализа.

### Шаг 00.1: Извлечение текста
- Проверить наличие `_output/extracted_text.txt`
- Если нет — запустить: `python process_project.py projects/<name>`

### Шаг 00.2: ОБЯЗАТЕЛЬНАЯ нарезка тайлов
**КРИТИЧЕСКОЕ ПРАВИЛО: Тайлы ОБЯЗАТЕЛЬНЫ. Без тайлов аудит НЕ ЗАПУСКАЕТСЯ.**

Проверить наличие PNG-файлов в `_output/tiles/page_*/`:
```bash
# Проверка — есть ли тайлы
ls projects/<name>/_output/tiles/page_*/
```

Если тайлов НЕТ или папка `tiles/` пуста:
1. Запустить: `python process_project.py projects/<name>`
2. Скрипт автоматически определит чертёжные страницы по размерам PDF
3. Если `tile_config` в `project_info.json` был пуст — скрипт заполнит его автоматически
4. Дождаться завершения нарезки
5. Убедиться что в `_output/tiles/` появились подпапки с PNG-файлами

**ЗАПРЕЩЕНО переходить к Этапу 01 если тайлы не созданы!**

## ЭТАП 01 — АНАЛИЗ ТЕКСТА PDF

### Выбор текстового источника

**MD-файл:** `{MD_FILE_PATH}`

Если путь к MD-файлу НЕ "(нет)":
1. **ЧИТАТЬ MD-файл как ОСНОВНОЙ текстовый источник** — он содержит:
   - Структурированный текст каждой страницы (`## СТРАНИЦА N`)
   - Таблицы в формате Markdown (спецификации, ведомости — точнее чем PyMuPDF)
   - Описания чертежей (блоки `IMAGE:` — тип, оси, сущности)
   - ~2x больше информации чем extracted_text.txt
2. `extracted_text.txt` использовать как **дополнительный источник** для перекрёстной проверки
3. При расхождениях между MD и extracted_text.txt — сверяться с тайлами (PDF — источник истины)

Если MD-файл отсутствует ("(нет)"):
- Читать `_output/extracted_text.txt` как единственный текстовый источник (как раньше)

## ЭТАП 02 — АНАЛИЗ ТАЙЛОВ ЧЕРТЕЖЕЙ

**ВАЖНО: Этот этап выполняется ВНЕШНИМ скриптом (run_tile_audit.ps1).**
**В основной сессии аудита НЕ НУЖНО самостоятельно читать тайлы.**

### Что делать:

1. Проверить наличие `_output/02_tiles_analysis.json`
2. Если файл **СУЩЕСТВУЕТ** → этап 02 уже выполнен, перейти к этапу 03
3. Если файла **НЕТ** → попробовать выполнить анализ тайлов:
   a. Проверить наличие `_output/tile_batches.json`
   b. Если нет — запустить: `python generate_tile_batches.py projects/<name>`
   c. Вывести сообщение: "Для полного анализа тайлов (100% покрытие) запустите:
      `.\run_tile_audit.ps1 -Project <name>`"
   d. Если пользователь не возражает — выполнить анализ ключевых тайлов самостоятельно
      (но предупредить что покрытие будет неполным)

### Почему НЕ нужно читать тайлы в основной сессии:

Одна сессия Claude может прочитать ~6-24 тайлов из 100-600+.
Это даёт покрытие **3-5%** — теряется до 60% замечаний.

Пакетный анализ (run_tile_audit.ps1) разбивает тайлы на пакеты по ~10 шт,
запускает отдельную Claude-сессию для каждого пакета и сливает результаты.
**Покрытие: 100%.**

## ПРАВИЛА JSON

1. Каждый JSON писать через инструмент Write — не выводить в чат
2. Формат даты: ISO 8601 — "2026-02-26T23:55:00"
3. ID: T-001..999 (текст), G-001..999 (чертежи), F-001..999 (финал)
4. При ответах на вопросы — читать 03_findings.json, не весь контекст
5. **Тайлы ОБЯЗАТЕЛЬНЫ** — если `tiles/` пуста, запустить `process_project.py` ПЕРЕД этапом 01
6. При пакетном аудите нескольких проектов — нарезать тайлы для ВСЕХ проектов без исключений

## Входные данные
Проекты находятся в папке: `projects/`

Для каждого проекта:
- `projects/<name>/document.pdf`               — оригинал (источник истины)
- `projects/<name>/_output/extracted_text.txt` — текст PDF
- `projects/<name>/_output/tiles/`             — тайлы чертежей
- `norms_reference.md`                         — нормативная база

## Задача
Выполни полный технический аудит КАЖДОГО проекта из папки `projects/`.

### Шаги для КАЖДОГО проекта (без пропусков):

**1. ИНИЦИАЛИЗАЦИЯ + ПОДГОТОВКА ДАННЫХ (ОБЯЗАТЕЛЬНО)**
- Прочитать `projects/<name>/project_info.json` — получить название, раздел, конфигурацию
- Проверить наличие `_output/extracted_text.txt`, если нет — запустить:
  `python process_project.py projects/<name>`
- **ОБЯЗАТЕЛЬНО** проверить наличие тайлов в `_output/tiles/`:
  - Если тайлов нет — запустить `python process_project.py projects/<name>`
  - Скрипт автоматически определит tile_config если он пуст
  - **НЕ ПРОДОЛЖАТЬ** аудит пока тайлы не будут созданы
  - Проверить результат: в `_output/tiles/` должны быть подпапки `page_XX/` с PNG

**2. ТЕКСТОВЫЙ АНАЛИЗ**
- Прочитать `_output/extracted_text.txt` (все страницы)
- Проверить нормативные ссылки в ПЗ на актуальность (по `norms_reference.md`)
- Проверить расчёт нагрузок, выбор кабелей, защитных аппаратов

**3. АНАЛИЗ ЧЕРТЕЖЕЙ** (по карте тайлов из project_info.json)
- Читать тайлы последовательно: все блоки страниц из tile_config
- Для каждого тайла фиксировать наблюдения

**4. ФОРМИРОВАНИЕ ЗАМЕЧАНИЙ**
Таблица:

| № | Категория | Лист/стр | Суть замечания | Норма | Рекомендация |
|---|-----------|----------|----------------|-------|--------------|

Категории:
- 🔴 КРИТИЧЕСКОЕ — нельзя строить/сдавать без исправления (ошибки расчёта, отменённые нормы)
- 🟠 ЭКОНОМИЧЕСКОЕ — лишние затраты, завышение объёмов, неоптимальный выбор
- 🟡 ЭКСПЛУАТАЦИОННОЕ — будущие проблемы в работе (ложные срабатывания, износ)
- 🔵 РЕКОМЕНДАТЕЛЬНОЕ — опечатки, неактуальные редакции, мелкие несоответствия
- ⚪ ПРОВЕРИТЬ ПО СМЕЖНЫМ — требует данных от смежных разделов или подрядчика

**5. СОХРАНЕНИЕ РЕЗУЛЬТАТА**
Сохранить в: `projects/<name>/_output/audit_results_YYYYMMDD.md`

Файл должен начинаться с:
```
# Аудит проекта: <name>
Дата: <дата>
Источники: PDF (стр. X), тайлы (N блоков)
```

## Тип вывода
- Отдельный файл audit_results_*.md для каждого проекта
- В конце — общее резюме по всем проектам
